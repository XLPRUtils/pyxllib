# pyxllib 生态重构指南：从“厚重封装”到“轻量标准”

## 1. 重构愿景 (Vision)
`pyxllib` 曾是高度集成的开发框架，封装了大量便利功能。但随着 Python 标准库的进化和项目复杂度的增加，过度封装带来了维护负担和学习成本。
本次重构的核心目标是：**去肥增瘦，回归标准**。
我们要将 `pyxllib` 从一个“替代标准库的框架”转变为一个“补充标准库的工具集”。

## 2. 依赖拓扑与连锁反应 (Dependency Chain)
重构涉及三个层级的仓库，修改必须严格遵循依赖顺序，处理连锁反应：

1.  **底层核心**: `pyxllib` (`src/pyxllib`)
    *   *角色*: 基础工具库。
    *   *动作*: 修改/移除底层类，提供迁移方案。
2.  **中间件**: `pyxlpr` (`src/pyxlpr`)
    *   *角色*: AI/CV 领域的扩展库，强依赖 `pyxllib`。
    *   *动作*: 适配 `pyxllib` 的变更，替换旧 API 调用。
3.  **业务应用**: `xlproject` (`src/xlsln`)
    *   *角色*: 最终业务逻辑。
    *   *动作*: 修复因底层变动导致的业务代码崩溃。

**AI 注意**: 你在修改 `pyxllib` 时，**必须** 意识到这会破坏 `pyxlpr` 和 `xlproject`，并负责修复它们。

## 3. 核心重构原则 (Principles)

### 3.1 拥抱标准库 (Embrace Standard Lib)
*   优先使用 Python 原生标准库（如 `pathlib`, `subprocess`），及常见的三方库如loguru。
*   **移除** 那些“仅仅为了省几个字符”而存在的 Wrapper 类（如已移除的 `File`，待移除的 `XlPath`）。
*   仅保留标准库确实缺失的高价值功能，并将其重构为独立的**纯函数** (Pure Functions)，而非复杂的类继承。

### 3.2 思想沉淀 (Legacy Wisdom Preservation)
旧代码虽然要被移除，但其中蕴含了多年的工程智慧（如：为了解决什么痛点？有什么独特的处理逻辑？）。
*   **在删除任何核心组件前**，必须分析其设计思想。
*   将这些“好的思想方法”总结记录到 `docs/legacy_wisdom.md`（或类似文档）。
*   **价值**: 即使代码没了，解决问题的思路要留给未来借鉴。

### 3.3 原子化推进 (Atomic Steps)
*   不要试图一次性重构整个库。
*   每次选择一个具体组件（例如 `XlPath`，或 `XlLog`），按“分析 -> 沉淀 -> 替换 -> 移除”的完整闭环执行。

## 4. 标准作业流程 (SOP for AI)

当指派你进行某个组件（如 `XlPath`）的重构时，请遵循以下步骤：

1.  **Analyze (分析)**: 阅读旧组件代码，理解它比标准库多做了什么（例如：自动 `mkdir`，自动编码检测）。
2.  **Document (沉淀)**: 在 `docs/` 下更新总结文档，记录该组件的独到之处。
3.  **Refactor (重构)**:
    *   在 `pyxllib` 中提供轻量级的替代方案（如工具函数）。
    *   在 `pyxllib` 内部替换所有引用。
4.  **Propagate (传播)**:
    *   搜索 `pyxlpr` 中所有引用，进行替换和适配。
    *   搜索 `xlproject` 中所有引用，进行替换和适配。
5.  **Clean (清理)**: 确认所有依赖方都已修复后，安全删除旧组件代码。
